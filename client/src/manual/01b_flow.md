# プロジェクトフロー一覧

**プロジェクトフロー**は、複数のテーマをどの順序・条件で進行させるかを決めるパターンです。

← [New Session の設定に戻る](page:setup)

---

## フロー一覧

### ウォーターフォール
テーマを順番に1つずつ実行します。最もシンプルな基本パターンです。

- **特徴**: テーマ1が完了したらテーマ2、という直線的な進行
- **向いている場面**: 多くのケースではこれで十分。迷ったらウォーターフォールを選ぶ
- **例**: テーマ1「課題整理」→ テーマ2「解決策立案」→ テーマ3「アクションプラン作成」

---

### ステージゲート
各テーマ完了後にゲートキーパーが品質チェックを行い、不合格なら差し戻します。

- **特徴**: 品質基準を満たすまで次のテーマに進まない
- **向いている場面**: 各ステージで明確な合格基準がある時、手戻りを防ぎたい時
- **例**: テーマ1完了 → ゲートキーパーが評価 → 合格なら次へ、不合格ならテーマ1に差し戻し

---

### アジャイル/スプリント
全テーマをひと通り実行し、完成度を評価して満足するまで繰り返します。

- **特徴**: 全体を繰り返すことで、全テーマを均等に改善できる
- **向いている場面**: 反復的に全体を磨き上げたい時
- **例**: テーマ1→2→3を1スプリントとして実行 → 評価 → 不満足なら再スプリント

---

### 条件分岐/ツリー
テーマの結論に応じて、次に実行するテーマを動的に選択します。

- **特徴**: 議論の結果によって異なる道筋を進む
- **向いている場面**: 「A案なら次はX、B案なら次はY」というように結論によって方向性が変わる時
- **例**: テーマ1の結論が「新規開発」→ テーマ2A「開発計画」、「既存改善」→ テーマ2B「改善計画」

---

### V字型
全テーマを順番に実行した後、逆順でレビューを行います。

- **特徴**: 前半（要件・設計）と後半（実装・検証）の整合性を確認できる
- **向いている場面**: 要件定義から実装まで段階的に進め、最後に全体の整合性を確認したい時
- **例**: テーマ1「要件」→ テーマ2「設計」→ テーマ3「実装」→ テーマ2'「設計レビュー」→ テーマ1'「要件整合確認」

---

### ゲーム理論/対立型
提案陣営と批判陣営が議論し、合意形成者が最終案を導きます。

- **特徴**: 対立構造を作ることで、より深い議論が生まれる
- **向いている場面**: 対立する視点を持つ複数案を比較検討したい時
- **例**: 提案チームが案を提示 → 批判チームが反論 → 合意形成者が最終案をまとめる

---

### ブラックボード型
コーディネーターが共有の状態（ブラックボード）を管理し、次の担当を動的に指名します。

- **特徴**: 全エージェントが共有情報を参照・更新しながら協調して作業できる
- **向いている場面**: 複数のエージェントが互いの作業に依存する複雑な協調作業
- **例**: コーディネーターが進捗状況を見て「今は法務レビューが必要」と判断して指名

---

### トーナメント/進化型
同じテーマを複数のグループで並列実行し、審査員が最良の結果を選択します。

- **特徴**: 異なるアプローチを同時に試して最良を採用できる
- **向いている場面**: 複数のアプローチを比較したい時、ベストプラクティスを見つけたい時
- **例**: グループA・B・Cが同じテーマで議論 → 審査員が最良の議論を選択して次のテーマへ

---

## 選び方のまとめ

| やりたいこと | おすすめのフロー |
|---|---|
| とりあえず順番に実行したい | ウォーターフォール |
| 品質基準をクリアしてから次へ進みたい | ステージゲート |
| 全体を繰り返し磨き上げたい | アジャイル/スプリント |
| 結論によって次のテーマを変えたい | 条件分岐/ツリー |
| 要件と実装の整合性を最後に確認したい | V字型 |
| 対立する複数案を徹底的に比較したい | ゲーム理論/対立型 |
| 複雑な協調作業を柔軟に進めたい | ブラックボード型 |
| 複数アプローチを並列で試したい | トーナメント/進化型 |
